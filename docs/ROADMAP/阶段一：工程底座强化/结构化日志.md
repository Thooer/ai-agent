# 结构化日志

每次请求记录 token 用量、耗时、工具调用结果，输出 JSON 格式，支持后续接入日志收集或可观测平台。

---

## 目标

面试能回答："你怎么观测系统？出了问题怎么排查？"

结构化日志的价值：每条日志是 JSON，可以直接用 `grep` / `jq` 过滤，也可以接 ELK / Loki 等平台做聚合。

---

## 方案选型

用标准库 `logging` + JSON formatter，不引入 `structlog`。

理由：`structlog` 功能更强，但当前需求只是 JSON 格式化输出，标准库足够，少一个依赖，配置也更直观。

---

## 日志字段设计

每条请求日志包含：

```json
{
  "timestamp": "2026-05-12T10:30:00Z",
  "level": "INFO",
  "request_id": "uuid",
  "user_id": "uuid",
  "conversation_id": "uuid",
  "event": "chat_complete",
  "duration_ms": 1234,
  "prompt_tokens": 512,
  "completion_tokens": 128,
  "total_tokens": 640,
  "model": "glm-5.1",
  "status": "completed"
}
```

失败时额外记录：

```json
{
  "status": "failed",
  "error_type": "TimeoutError",
  "error_message": "LLM request timed out after 60s"
}
```

---

## 实现方式

### JSON Formatter

在 `core/logging.py` 实现一个 `JsonFormatter`，继承 `logging.Formatter`，`format()` 方法将 LogRecord 序列化为 JSON 字符串。

### request_id 传递

在 FastAPI middleware 里为每个请求生成 `request_id`（UUID），存入 `contextvars.ContextVar`，日志里直接读取，无需手动传参。

### 日志记录位置

在 `routers/ai_chat.py` 的 `event_generator` 里，流结束（`done` 或 `error`）时记录一条完整日志。token 用量从 LLM API 响应的 `usage` 字段读取（BigModel API 在最后一个 chunk 里返回 usage）。

---

## 日志级别约定

```
INFO：正常请求完成，记录 token 用量和耗时
WARNING：重试发生，记录重试次数和原因
ERROR：请求最终失败，记录错误类型和 message
DEBUG：开发阶段详细信息，生产不输出
```

---

## 实现文件

| 文件 | 变更 |
|------|------|
| `core/logging.py` | 新建，JsonFormatter + 日志初始化函数 `setup_logging()` |
| `core/middleware.py` | 新建，request_id 生成中间件，写入 ContextVar |
| `main.py` | 启动时调用 `setup_logging()`，注册 request_id 中间件 |
| `routers/ai_chat.py` | 流结束时记录结构化日志 |
| `services/llm.py` | 重试时记录 WARNING，返回 usage 信息 |

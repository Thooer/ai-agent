# BM25 索引

> **状态：✅ 已完成**

关键词检索，与向量检索互补（向量擅长语义，BM25 擅长精确词匹配）。

---

## 方案

`rank-bm25` 库，进程内索引，不上 Elasticsearch。

理由：当前文档量小，内存够用；面试能讲"做了显式的工程取舍，知道 ES 的适用场景"。

---

## 索引生命周期

```
应用启动
  └── 从数据库加载所有 chunks 内容，按 user_id 分组建索引

文档新增/删除
  └── 重建该 user_id 的 BM25 索引

检索时
  └── 直接查对应 user_id 的内存索引
```

---

## 内存结构

```python
# 全局单例，进程内维护
_bm25_indexes: dict[UUID, BM25Index] = {}

@dataclass
class BM25Index:
    bm25: BM25Okapi
    chunk_ids: list[UUID]   # 与 BM25 内部 doc 顺序对齐
    built_at: datetime
```

`chunk_ids` 列表与 BM25Okapi 内部文档顺序严格对齐，根据 BM25 返回的 doc 下标取到对应的 chunk_id。

## 分词

中英文混合内容，分词用空格 + 标点切分（简单方案），不引入 jieba。

理由：embedding-3 已经处理中文语义，BM25 主要捡精确关键词（专有名词、代码片段、英文术语），简单分词对这类场景足够。面试能讲这个取舍。

## 索引重建触发点

- 应用启动 lifespan：`await rebuild_bm25_index(user_id)` for all users
- ingestion complete_task 后：`await rebuild_bm25_index(user_id)`
- 文档删除后：`await rebuild_bm25_index(user_id)`

重建是异步操作，重建期间检索仍使用旧索引（stale read 可接受）。

---

## 冷启动与空索引

- 数据库无 chunks 时，返回空索引，检索返回空列表
- 首次 ingestion 完成后触发重建

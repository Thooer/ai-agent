# 阶段二：RAG 核心 — 概览与推进顺序

目标：做出知识库问答能力，这是简历第一个核心模块。

---

## 推进顺序

| # | 任务 | 说明 |
|---|------|------|
| 1 | 结构化日志 | ingestion pipeline 最难排查，没日志就是盲盒，先于 RAG 补上 |
| 2 | pgvector + RAG 表迁移 | documents / chunks 两张表，pgvector 扩展初始化 |
| 3 | JWT 鉴权 | 文档解析接口是第一个需要 user_id 的接口，在此之前做好，接口签名一次写对 |
| 4 | 文档解析 | PDF / Markdown / TXT → 纯文本 |
| 5 | Chunking | 固定窗口 + overlap，tiktoken 计 token 数 |
| 6 | Embedding 批处理 + 缓存 | 智谱 embedding-3，1024 维，走 Redis 缓存 |
| 7 | Ingestion background task + Redis 进度 | 耗时操作异步化，task_state 报进度 |
| 8 | BM25 索引 | rank-bm25，进程内，启动时加载，文档更新后重建 |
| 9 | 向量检索 | pgvector cosine，top-10 |
| 10 | RRF hybrid search | 向量 + BM25 各 top-10，RRF k=60，final top-5 |
| 11 | Chat 接入 RAG context | 检索结果注入 system prompt，返回引用来源 |

---

## 核心参数（全部配置化）

| 参数 | 默认值 |
|------|--------|
| chunk_size | 512 token |
| chunk_overlap | 64 token |
| embedding_dim | 1024 |
| vector_top_k | 10 |
| bm25_top_k | 10 |
| rrf_k | 60 |
| final_top_k | 5 |

---

## 多租户设计

- `documents` / `chunks` 都带 `user_id`
- 所有检索、ingestion 必须按 `user_id` 隔离
- JWT 未完成前用 dev user（固定 UUID）过渡，JWT 上线后只改 user_id 来源，业务逻辑不动
